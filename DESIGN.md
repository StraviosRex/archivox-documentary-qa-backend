# Design Document: Documentary Q&A Backend

## Overview

Archivox answers natural-language questions using evidence from a documentary transcript. When the service starts, it divides the transcript into timestamped chunks, creates embeddings for them, and stores them in a searchable index. For each question, a hybrid retrieval pipeline gathers candidate passages, and a cross-encoder then reranks and filters them before the selected evidence is sent to the LLM.

Answers are generated by the LLM, but source references are not. Timestamps and excerpts are taken directly from the retrieved chunks, keeping citations traceable and preventing the model from inventing them.

## 1. Chunking Strategy

The transcript alternates between `HH:MM:SS` timestamps and spoken-text blocks. Archivox parses these into `(timestamp, text)` pairs and combines every two consecutive segments into a chunk, with one segment overlapping between neighbouring chunks. This produces approximately 259 chunks, generally containing 200–350 words.

I chose timestamp-based grouping because the transcript already provides useful speech boundaries and reliable source locations. A fixed character split could cut through a sentence or separate a claim from the explanation that follows it. The one-segment overlap reduces the chance of losing information near a chunk boundary and gives the retriever more than one opportunity to surface a passage that spans adjacent segments.

Each chunk stores its full text, start and end timestamps, and the indexes of its source segments. The API builds source references directly from this metadata, so every returned time range maps back to a real location in the transcript.

The chunker also attaches configurable metadata tags from `config/retrieval.yaml`. For this transcript the active field is `era`, but the implementation treats metadata fields generically. This keeps the Python code from being tied to one documentary; another corpus could define fields such as `speaker`, `topic`, or `chapter` in YAML and reuse the same chunking and filtering flow.

## 2. Retrieval

**Embedding model.** Chunk text is embedded locally with `all-MiniLM-L6-v2` from sentence-transformers, producing 384-dimensional vectors. The question is embedded with the same model, and semantic similarity is measured using cosine distance.

**Vector store.** Embeddings and chunk metadata are stored in a persistent ChromaDB collection using an HNSW index. The application records the transcript hash, embedding model, chunking settings, retrieval config hash, and index version alongside the index. If any of these inputs change, the index is rebuilt automatically.

**Metadata filtering.** Query patterns in `config/retrieval.yaml` can detect metadata labels in the question. Matching labels are passed to Chroma as a `where` filter before dense retrieval, so irrelevant chunks can be pruned before reranking. The current config uses this for historical eras, while still keeping the mechanism generic.

**Hybrid candidate retrieval.** Dense vector search works well for semantic similarity, but it can miss passages where exact wording matters, especially names, numbers, and specific terms. To compensate for this, Archivox also applies lexical scoring across the transcript. This considers query-term coverage, exact proper names, capitalized entities, numerical matches, configured concept aliases, and literal query terms.

The dense and lexical results are merged and deduplicated. This gives the reranker a broader candidate pool that includes both semantically related passages and exact-term matches.

**Cross-encoder reranking.** Each candidate is then scored with `cross-encoder/ms-marco-MiniLM-L-6-v2`. Unlike the embedding model, which represents the question and chunk separately, the cross-encoder evaluates them together. This allows it to judge their relationship more precisely.

When reranking is enabled, the cross-encoder score becomes the final relevance filter. Candidates scoring below `−4.5` are removed. This threshold was chosen from the score distributions observed during regression testing: relevant passages consistently appeared above it, while weak matches caused by common vocabulary generally scored much lower.

If reranking is disabled, the system falls back to lexical strength and cosine-distance filtering. This path is faster, but testing showed that it was less reliable for questions covering several topics.

**Context-aware source selection.** The strongest cross-encoder result becomes the primary evidence anchor. Supporting evidence is then selected in one of two ways:

* Local mode adds neighbouring transcript chunks when they provide useful surrounding context.
* Diverse mode favours passages that cover different parts of a broad or multi-topic question.

A neighbouring chunk may be included for continuity even if it did not independently pass the cross-encoder threshold. Archivox returns up to three sources. The first represents the strongest relevance anchor, while the remaining sources provide either supporting context or additional topic coverage. They are therefore ranked by overall usefulness to the question rather than being presented as a strict list of the three highest raw cross-encoder scores.

## 3. Prompt Construction

The LLM receives only the selected transcript passages, each labelled with its timestamp range, followed by the user's question. The full transcript is never placed into the prompt.

The system prompt instructs the model to answer only from the supplied excerpts, keep the response concise, avoid unsupported claims, and use a fixed refusal when the evidence is insufficient.

```text
System:
Answer only from the provided transcript excerpts.
If they do not contain enough information, return the fixed refusal phrase.
Use 2–4 sentences of plain prose without timestamps or Markdown.

User:
[Excerpt 1 | 00:10:33 - 00:13:21]
Boracic acid was a component of a product called borax, used during the
Victorian period to neutralise the acid in sour milk...

[Excerpt 2 | 00:13:21 - 00:16:14]
The real problem is it doesn't get rid of the bacteria, the underlying
cause of the acid...

Question:
What was borax used for in Victorian milk?
```

The backend creates the `sources` array separately from the generated answer. The LLM is not asked to invent timestamps, excerpts, or chunk identifiers. This separation keeps the answer natural while ensuring that every displayed citation comes from evidence the retriever actually selected.

## 4. LLM Provider Configuration

LLM backends are defined as named profiles in `config/models.yaml`. Each profile specifies the provider, model, base URL, and the environment variable containing its API key. The active profile can be selected through `LLM_PROFILE` or overridden in an individual `/ask` request.

Groq, Gemini, OpenRouter, and Ollama are supported through OpenAI-compatible chat-completion interfaces. Ollama was also tested as a fully local option and produced grounded answers across the evaluation categories, including a correct refusal for an out-of-scope question.

The default Groq configuration averaged approximately 3.47 seconds end to end with reranking enabled. Local Ollama inference completed in roughly 9 seconds. Both remain comfortably within the 30-second response-time requirement.

## 5. API Response Construction

The API returns the generated answer, selected sources, and model metadata. The `sources` array is constructed by the backend from retrieved chunks, not by the LLM. This means timestamps, excerpts, and model/provider fields are deterministic parts of the response.

If the answer uses the fixed insufficient-evidence refusal, the backend returns an empty `sources` list. I also added `tests/audit_sources.py` as a separate grounding check: it verifies that returned timestamps and excerpts can be traced back to `data/transcript.txt`. It does not judge semantic relevance; it only checks that citations are real.

## 6. What I Would Improve Given More Time

* **BM25 lexical retrieval:** Replace the hand-tuned lexical scoring with BM25 to gain term-frequency saturation and document-length normalization.
* **Bounded candidate fusion:** Use rank fusion to combine dense and lexical rankings before cross-encoder reranking. This could reduce latency without weakening multi-topic coverage.
* **Dynamic chunking:** Detect topic shifts and build variable-length chunks around content boundaries rather than always combining a fixed number of transcript segments.
* **Streaming:** Add optional token streaming while preserving the existing source-reference structure.
