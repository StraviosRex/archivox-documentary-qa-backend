# Design Document: Documentary Q&A Backend

Archivox is a retrieval-augmented Q&A backend that answers natural-language questions using only selected transcript excerpts, returning answers with timestamped sources.

## 1. Chunking Strategy

The transcript is formatted as alternating `HH:MM:SS` timestamp lines and spoken-text blocks. On startup, Archivox parses these into timestamped segments and combines every two consecutive segments into an overlapping chunk, with one segment shared between neighbouring chunks. This keeps speech boundaries intact, avoids splitting a claim from its explanation, and gives the retriever a second chance to find information near a boundary. Each chunk stores its text, start and end timestamps, and configurable metadata fields defined in `config/retrieval.yaml`.

## 2. Retrieval Strategy

Chunks are embedded locally with `all-MiniLM-L6-v2` and stored in a persistent ChromaDB collection using cosine distance. At query time, configured metadata labels from `config/retrieval.yaml` can become Chroma filters, then retrieval combines dense vector search with lexical scoring that catches exact names, numbers, and configured aliases. Dense and lexical candidates are merged and deduplicated into a reranking pool, where `cross-encoder/ms-marco-MiniLM-L-6-v2` scores each question-chunk pair directly. Candidates below the cross-encoder threshold are filtered out.

After filtering, source selection depends on question shape. Local questions use the strongest chunk as an anchor and may include neighbouring transcript chunks for context. Broad or multi-topic questions use a diverse selection mode that favours chunks covering different query terms or transcript regions. The API returns up to three sources, ordered by usefulness to the question.

## 3. Prompt Construction

The prompt contains only the selected excerpts, each labelled with its timestamp range, followed by the user question. The full transcript is never sent to the LLM. The system prompt instructs the model to answer only from the provided excerpts, avoid unsupported claims, and use a fixed refusal when evidence is insufficient. Source timestamps and excerpts in the API response are constructed by the backend from selected chunks, not generated by the model. LLM backends are configured as profiles in `config/models.yaml`, supporting Groq, Gemini, OpenRouter, and Ollama.

## 4. Improvements Given More Time

I would split the retrieval module into smaller components for indexing, metadata filtering, scoring, reranking, and source selection. I would also replace the hand-tuned lexical scoring with BM25 or rank fusion, explore topic-aware chunking instead of fixed two-segment windows, and add optional response streaming while preserving the backend-owned source citation flow.
